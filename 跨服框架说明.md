## 4. 最关键的几个核心概念

### 4.1 会话锁

同一个玩家同一时刻只能被一台服务器持有写权限。

比如玩家从 A 服切到 B 服：

- A 服先占有玩家会话
- 玩家切服时 A 服释放会话
- B 服重新申请会话

如果 B 服申请不到，说明玩家数据可能还在另一台服处理中，这时会阻止玩家登录，避免两台服同时写玩家数据。

当前实现中：

- 数据库中的 `player_session` 是唯一权威来源
- Redis 中的 session 消息用于让其他节点感知会话 acquire / release 事件
- 远端会话感知不会替代数据库锁判断

### 4.2 命名空间

命名空间是为了支持多个模块共享同一套跨服底层。

当前内置已使用的命名空间包括：

- `core.player`
- `core.global`
- `session`
- `economy`
- `inventory`
- `enderchest`
- `player-state`
- `homes`
- `auth.profile`
- `auth.ticket`

### 4.3 MySQL 与 Redis 的职责

- MySQL：最终权威数据源
- Redis：广播同步消息与失效通知

保存数据后会自动发出同步消息；远端节点收到后会记录失效信息，或交给同步监听器处理。

### 4.4 玩家认证与会话的关系

- `SessionService` 负责跨服唯一写会话
- `AuthService` 负责玩家是否已认证
- 进入服务器时先拿会话，再处理登录态
- 跨服免重登只影响认证，不会绕过会话锁

## 5. 当前数据表

插件启动后会自动创建这些表：

### 5.1 `player_session`

保存玩家当前会话锁：

- `player_uuid`
- `current_server`
- `session_id`
- `lock_until`
- `last_seen`

### 5.2 `player_snapshot`

保存玩家级别的数据快照：

- `player_uuid`
- `namespace`
- `payload`
- `version`
- `updated_at`

当前已经使用到这些命名空间：

- `economy`
- `inventory`
- `enderchest`
- `player-state`
- `homes`
- `auth.profile`
- `auth.ticket`
- `teleport.handoff`

### 5.3 `global_snapshot`

保存全局共享数据：

- `namespace`
- `data_key`
- `payload`
- `version`
- `updated_at`

### 5.4 `player_identity`

保存玩家 UUID 与名称映射：

- `player_uuid`
- `player_name`
- `updated_at`

### 5.5 `economy_transaction`

保存经济流水：

- `id`
- `player_uuid`
- `player_name`
- `transaction_type`
- `amount`
- `balance_after`
- `source`
- `created_at`

### 5.6 `node_status`

保存节点心跳：

- `server_id`
- `cluster`
- `last_seen`
- `updated_at`

## 6. 当前业务模块

## 6.1 economy

`economy` 已接在统一的快照与广播流程上：

- 查询余额时读取 `player_snapshot` 中 `economy` 命名空间的数据
- 金币变化时写回 `player_snapshot`
- 保存后自动触发跨服广播
- 金额统一按两位小数处理
- 提供 Vault Economy 兼容层

## 6.2 inventory / enderchest

- 玩家进服后读取 `inventory` 和 `enderchest` 快照
- 玩家退出或被踢出时保存主背包和末影箱
- 实际存储仍复用 `player_snapshot`

## 6.3 player-state

- 玩家进服后读取 `player-state` 快照
- 玩家退出、被踢出、死亡、重生后保存玩家基础状态
- 实际存储仍复用 `player_snapshot`

## 6.4 homes

### 当前能力

- 玩家 homes 数据跨服共享
- 数据存储在 `player_snapshot` 的 `homes` 命名空间
- 支持 `/home`、`/homes`、`/sethome`、`/delhome`、`/setdefaulthome`
- `/homes` 现在会打开菜单 UI
- 数据结构已包含 `serverId`

### 当前边界

- 如果 home 位于其他服务器，会委托 `CrossServerTeleportService` 准备 handoff
- homes 继续只负责目标位置，不直接处理 proxy transfer

### homes 数据结构

每个 home 包含：

- `name`
- `serverId`
- `world`
- `x`
- `y`
- `z`
- `yaw`
- `pitch`
- `createdAt`
- `updatedAt`

快照还包含：

- `defaultHome`
- `homes`

## 6.5 auth

### 当前能力

- 支持注册、登录、修改密码
- 支持聊天栏输入密码完成登录
- 支持短时跨服免重登
- 支持未登录行为限制
- 玩家登录过程中显示 Title、ActionBar、BossBar

### auth 数据分层

#### A. 长期资料 `auth.profile`

存储：

- `passwordHash`
- `salt`
- `registeredAt`
- `lastLoginAt`
- `lastLoginServer`
- `loginTicketVersion`

#### B. 短期票据 `auth.ticket`

存储：

- `tokenId`
- `issuedAt`
- `expiresAt`
- `issuedServer`
- `version`

### 当前认证体验

- 未注册玩家：提示注册
- 已注册未登录玩家：提示登录
- 登录成功后关闭 BossBar 并显示成功 Title
- 短时间跨服切换时，如果 ticket 未过期，则自动通过认证

### 当前限制行为

未认证玩家会被限制：

- 移动
- 打开背包
- 点击背包
- 丢弃物品
- 拾取物品
- 交互
- 执行非白名单命令

## 6.6 teleport.handoff

### 当前能力

- 已新增独立 `teleport` 模块
- 可为玩家写入一次性 `teleport.handoff` 快照
- source 服可准备 session baton-pass，并把 transfer token 写入 handoff
- 已接入基于 proxy plugin messaging 的真实切服网关
- 目标服在玩家到达后可尝试 consume handoff 并执行落点传送
- `/crossserver transfer info <player>` 与 `/crossserver transfer menu <player>` 可读取最近一次 handoff 快照，直接排查 requestId、状态、gateway 发送、arrival consume、prepared transfer、session baton 与 failure reason
- `homes` 的跨服 home 已改为委托 handoff 服务准备传送
- transfer 关键节点会显示 Title / ActionBar / Sound 反馈
- 已新增 `ServerTransferGateway` 抽象

### 当前边界

- 已完成 session baton-pass，因此目标服可在预登录阶段 claim 会话
- 已接入真实 proxy gateway，因此跨服 home 已可触发真实切服
- 当前仍未包含 ACK、自动重试、历史审计列表等更强恢复能力

## 7. 当前版本的工作方式

### 7.1 玩家进入服务器

玩家登录前先尝试申请玩家会话锁：

- 成功：允许进入
- 失败：阻止登录

玩家进入服务器后：

- 加载背包数据
- 加载玩家状态数据
- 加载 homes 数据
- 初始化登录状态

### 7.2 玩家在线期间

- 会定时刷新会话锁
- 会定时保存在线玩家的背包和状态数据
- 认证模块会周期性刷新登录提示 UI

### 7.3 玩家离线或被踢出

- 保存背包数据
- 保存玩家状态数据
- 保存 homes 数据
- 清理本地认证状态
- 释放会话锁

## 8. 其他插件怎么接入

推荐接入流程：

1. 通过 Bukkit `ServicesManager` 获取 `CrossServerApi`
2. 注册命名空间
3. 读取玩家数据或全局数据
4. 保存数据
5. 如有需要，注册同步监听器处理远端更新

当前可以直接参考：

- `economy`
- `homes`
- `auth`

这些模块都复用了统一的快照与广播机制。

### 9. 当前 handoff 基础设施

当前这一轮已经把跨服传送从“纯规划”推进到了真实可切服、可诊断的阶段。

已落地：

- 新增 `teleport.handoff` 命名空间
- 新增独立 `teleport` 模块
- `homes` 跨服目标会委托 `CrossServerTeleportService` 准备 handoff
- 目标服 `AsyncPlayerPreLoginEvent` 可通过 session baton-pass claim 会话
- 目标服 `PlayerJoinEvent` 后可消费 `PENDING` handoff 并执行落点 teleport
- `auth` 可在 handoff 准备阶段签发新的跨服 ticket
- 已新增 `ServerTransferGateway` 抽象
- 已接入基于 plugin messaging 的真实 proxy gateway
- 默认通过 `teleport.gateway.server-map` 将内部 `serverId` 映射到代理 backend 名称
- `/crossserver transfer <player>` 兼容旧诊断用法
- `/crossserver help` 与直接 `/crossserver` 会按当前权限显示可用命令
- `/crossserver transfer info <player>` 查看详细状态
- `/crossserver transfer clear <player>` 清理卡住的 transfer 状态
- `/crossserver transfer menu <player>` 打开管理员 transfer 菜单
- `/crossserver transfer history <player>` 查看玩家最近 transfer 历史
- `/crossserver transfer recent [page]` 打开带分页的 recent transfer 菜单
- `/crossserver transfer reconcile <player>` 触发一次保守 reconcile，补 ACK、补清理并收敛过期/残留状态
- `transfer.view` 负责文本诊断与历史查看，`transfer.menu` 负责 recent/menu 菜单入口，`transfer.reconcile` 负责保守修补动作
- `/crossserver auth inspect|invalidate|forcereauth <player>` 提供 auth 管理能力
- `CrossServerApi` 已可供其他插件发起指定玩家到指定服务器/世界/坐标的跨服传送
- transfer history 已新增 append-only 历史记录
- 已补 ACK / recovery 基础状态、prepared/baton 残留诊断与 recent -> detail -> history 三层运维菜单
- `/homes` 打开 homes 菜单，左键传送、右键设默认、Shift+右键删除

### 当前仍未完成的部分

- 未做自动重试
- 历史记录已支持 recent/history 与 append-only `transfer_history`，但仍未做更完整的独立审计表与更强观测面板
- 当前恢复仍以最小清理和人工诊断为主

### 后续推荐顺序

下一轮建议继续做：

1. ACK / source confirm
2. 自动恢复与补偿策略
3. 历史审计与更完整观测

## 11. 测试建议

### 11.1 单服验证

1. 启动服务器
2. 执行 `/crossserver status`
3. 执行 `/sethome main`
4. 执行 `/homes`
5. 执行 `/home main`
6. 新玩家进入服务器，检查注册提示 UI
7. 使用 `/register <password> <confirm>` 完成注册
8. 使用 `/changepassword <old> <new>` 验证改密
9. 执行一次 `/crossserver reload`
10. 再次验证 homes、auth、economy

### 11.2 多服验证

准备两台子服，共用同一个 MySQL 和 Redis。

重点验证：

- 节点状态是否正常
- A 服设置的 home，B 服是否可见
- `/homes` 是否能打开菜单，菜单内左键传送、右键设默认、Shift+右键删除是否按权限生效
- 指向其他服的 home 是否触发真实切服
- `/crossserver transfer info <player>` 与 `/crossserver transfer menu <player>` 是否能看到 requestId、状态推进、gateway 发送时间、arrival consume 时间、prepared transfer 状态与 failure reason
- 玩家到达目标服时是否能消费 `teleport.handoff` 并落到正确位置
- source/target 服 transfer Title / ActionBar / Sound 是否正常
- A 服登录后切到 B 服是否可免重登
- ticket 过期后是否重新要求登录
- 金币、背包、状态是否一致
- 节点离线状态是否按 `node.offline-seconds` 生效
